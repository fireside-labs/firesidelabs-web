HTTP Timeouts as Silent Benchmark Sabotage

How Fixed Timeouts Silently Tank Model Scores

A common pattern in LLM benchmarking: set a fixed HTTP timeout (e.g., 240 seconds) and call it done. This ruins your benchmarks.

A fixed 240-second timeout means models that respond in 2 seconds waste time, while models that need 45 seconds get killed. You measure timeout behavior, not model capability.

A model that scores 89.6% under fixed 240s timeout actually scores 95.1% when given an appropriate per-model timeout. The difference: 5.5 percentage points of performance artificially suppressed.

The Problem

Why It Happens

When running a benchmark, you set an HTTP socket timeout. If no response arrives within the timeout window, the request fails. A 240-second timeout seems safe — most responses come back faster. But:

Large models need more time. Gemma-4-31B running complex reasoning on HumanEval needs 45–60 seconds per question. At a 45-second timeout, 12 of 111 questions fail. Questions that timeout are counted as failures, just like wrong answers. But they're not model failures — they're infrastructure failures.

Concurrency adds delay. In production, running 25 models across 3 nodes with shared infrastructure creates queue buildup. Network jitter, inference queue delays, GPU context switching can push effective per-request latency past any fixed ceiling.

The Real Story

When we ran 25 models concurrently and hit 12 timeouts on Gemma-4-31B, the max observed latency in isolation was 44.8 seconds. So how did questions time out at 240 seconds? Because the 240-second timer started before the request even reached the GPU. Under concurrent load, the effective timeout was more like 45–60 seconds per question. The problem: we didn't know this was happening. Timeouts were silent.

The Solution: Dynamic Timeouts

Implement auto_timeout(model_params, benchmark_type) that scales timeout based on model size and task complexity.

def auto_timeout(model_params_b: int, benchmark_type: str) -> int: base_latency = { 'mmlu': 2.0, 'humaneval': 15.0, 'gsm8k': 8.0, } latency = base_latency[benchmark_type] * (model_params_b / 7.0) timeout = latency * 3.5 # safety margin return max(10, min(300, int(timeout)))

Examples: auto_timeout(7B, 'mmlu') = 7s, auto_timeout(31B, 'humaneval') = 232s

Practical Recommendations

1. Implement Dynamic Timeouts: Fixed timeouts are lies. Calculate based on model size and task complexity.

2. Make Timeouts Visible: Report in metadata: how many questions timed out, what percentage, what the score would be if timed-out questions were excluded.

3. Monitor for Timeouts: If >1% of questions timeout, increase timeout and re-run. Timeouts are a signal something is wrong.

4. Stress-Test at Realistic Load: Test benchmarks under realistic concurrency. Infrastructure contention will increase effective latency.

Fixed HTTP timeouts suppress reported accuracy by 5–15 percentage points on reasoning benchmarks. The fix is straightforward and costs nothing. Use dynamic timeouts. Report timeout incidents.
