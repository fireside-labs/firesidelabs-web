Reproducible Local LLM Benchmarking

A Practical Guide for Local Inference Teams

Local LLM benchmarking should be reproducible. Instead, most teams discover it isn't — not because models are unpredictable, but because the infrastructure underneath them is.

Run a benchmark on Monday, get results. Run it again on Friday. The numbers don't match. Not by a little. Sometimes by 5–10 percentage points. You blame the model. You blame the hardware. You run it three more times and now you have three different answers.

The problem isn't that models are unpredictable. The problem is that we're not measuring models — we're measuring infrastructure, and we don't have enough visibility into what's actually happening.

Without reproducible benchmarks, you can't compare models reliably, trust your optimizations, or debug failures. Everything becomes suspect.

This paper documents the reproducibility hierarchy — a five-level framework for understanding where drift comes from in local inference. We validate it empirically across a 3-node RTX 5090 cluster running 25 models, demonstrating reproducibility within ±0.5 percentage points across independent runs.

The techniques are straightforward: binary verification, per-host isolation, dynamic timeouts, per-question logging, and hybrid caching. They cost nothing to implement. They eliminate the silent failures that plague local-inference workflows.

The Reproducibility Hierarchy

Reproducibility isn't binary. It's a spectrum with five levels. Understanding which level you're at determines what else you can trust.

Level 1
Model File Identity

The same model name can hide multiple different files. SHA256 verification is non-negotiable. Without it, nothing downstream matters.

Level 2
Binary (Compilation) Parity

Two identical model files can produce different results if the inference engine changed. A compiler flag, library version, or bug fix can shift performance by 5–9 points.

Level 3
Hardware SKU Parity

Every chip has manufacturing variations. Run the same benchmark on multiple "identical" nodes and measure drift. Healthy parity is ±0.5pt. Larger drift signals something upstream is wrong.

Level 4
Silicon Consistency (Driver & Firmware)

GPU driver versions, BIOS, and thermal states all affect performance. A driver version difference alone can account for 0.8–1.2pt drift. Log these for every benchmark run.

Level 5
Environmental Consistency

Background processes, cache states, and system load introduce noise you can't eliminate — only measure. Run benchmarks on dedicated, quiet hardware. Log system metrics alongside results.

Why This Hierarchy Matters

Each level depends on the ones below it. If you haven't locked the binary (Level 2), measuring hardware drift (Level 3) is meaningless. If you haven't verified model files (Level 1), you don't know what you're testing. The hierarchy forces discipline.

The Foundry Harness: Putting It Into Practice

With the hierarchy in mind, we designed the Foundry harness — an inference orchestration framework that enforces reproducibility at every level.

Core Principles

Binary Locking: Every run freezes the exact binary used in its metadata. We maintain a CANONICAL_HASHES.json that marks specific builds as "golden" for cross-run comparison.

Per-Host Output Paths: Early versions used shared output directories. Multiple nodes writing to the same place caused silent race conditions — the last write won, and we lost data from earlier runs. Now each node writes to its own dated, host-specific directory.

Dynamic Per-Request Timeouts: A fixed timeout (240 seconds) is wrong for all models except one. Smaller models finish in 2 seconds. Large models need 45+ seconds. We implement auto_timeout(model_params, benchmark_type) that scales appropriately.

Per-Question JSONL Logging: Most benchmarking pipelines save results as bulk JSON or CSV. This loses information. We log every question with its full output, token counts, latency, and correctness judgment. This enables distillation, debugging, and analysis.

Hybrid Local Cache: Model files are large (4–30GB). Transferring across a network for every run is slow. We use local NVMe cache on each node, with checksums to verify cache hits. First run takes time to fetch. Subsequent runs start in seconds.

Empirical Validation: 3-Node Cluster

We validated the framework across a 3-node GPU cluster (Node-A, Node-B, Node-C) running 25 different models on three benchmarks: MMLU, HumanEval, and 500Q.

When drift exceeds ±0.5pt, investigation is warranted. We discovered that Node-C was running NVIDIA driver 591.55 while Node-A ran 596.21. Once the driver was updated, parity improved to ±0.3pt across the board.

Key Contributions

1. Dynamic Timeout Scaling

A fixed 240-second timeout suppresses Gemma-4-31B's HumanEval score from 95.1% to 89.6% (5.5pt loss) because 12 of 111 questions exceed the timeout window. Dynamic timeouts calibrated to model size recover the full capability.

2. Per-Question Logging

Instead of bulk JSON, we log each question with its full output, token counts, latency, and correctness judgment. This enables distillation (extract failed questions for retraining), debugging, and analysis. A dataset of 25 models × 5 benchmarks = 250K records at ~2KB each (500MB total). Trivial by modern standards. The value is immense.

3. Contamination Probing

We built the letter-shuffle contamination probe (see Paper 2) into the harness as a mandatory validation pass. Any model showing >5pt sensitivity to letter position is flagged as unreliable.

Practical Recommendations

1. Lock the Binary

Before you run any benchmark, freeze the exact version of your inference engine. Document it in metadata. If you upgrade, re-run benchmarks. Don't pretend old results are valid.

2. Use Per-Host Isolated Paths

If running across multiple machines, give each one its own output directory. Use date-based subdirectories. Never use shared directories with last-writer-wins semantics.

3. Implement Dynamic Timeouts

Fixed timeouts are lies. Calculate based on model size and task complexity. Report which questions timed out. If >1% of questions timeout, re-run with higher timeout.

4. Log Per-Question

Store the full details of every question: input, output, latency, reasoning trace, correctness. Not bulk summaries. Individual question records enable distillation and debugging.

5. Validate with Contamination Probes

Add the letter-shuffle probe to your pipeline as a mandatory validation pass. Flag any model showing ≥5pt sensitivity to letter position. These models are unreliable in production.

With proper infrastructure — binary locking, per-host isolation, dynamic timeouts, per-question logging, and contamination probing — local LLM benchmarking can be as reproducible as you need it to be.
